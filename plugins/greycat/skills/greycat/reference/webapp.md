# Adding a webapp to a GreyCat project

The one prescribed way to ship a web frontend with a GreyCat backend. It is opinionated on purpose: every
GreyCat webapp uses the same toolchain and layout, so any of them reads as "a classic GreyCat app." Deviate
only when a project has a concrete reason to.

The stack:

- **VitePlus** (`vp`) as the toolchain, building `app/` into `webroot/`.
- **MPA** (multi-page): each page is a real HTML file; the URL is the file path. No SPA router.
- **Lit** with **shadow DOM** (the default): one root element per page, plus a component for any view reused
  across pages. Each component owns its styles in a `static styles` block (a `css` tagged-template literal).
- **UI kit**: Web Awesome (`wa-*`) for every visual component - buttons, inputs, dialogs, tabs, cards, layout
  utilities, the `<wa-page>` shell, and the rest. Its `--wa-*` design tokens are the single source of truth for
  color, spacing, and type.
- **`app/theme.css`** for the brand: a small layer that overrides a few `--wa-*` tokens so every `wa-*` element
  adopts the app's colors. These custom properties inherit across the shadow boundary, so every component reads
  them with `var(...)`.
- **`@greycat/web/sdk`** for every backend call - and nothing else from `@greycat/web`. The SDK is a headless
  runtime (typed calls, `init`, `login`); it ships no components and no CSS. This makes **`greycat codegen ts`**
  mandatory.

**Web Awesome ships its own agent skills.** Once `@awesome.me/webawesome` is installed, read them before
building UI - they are the authoritative reference for component APIs, layout, and theming:

- `node_modules/@awesome.me/webawesome/dist/skills/webawesome/` - per-component reference (props, slots,
  events) and "which component do I pick" guidance.
- `node_modules/@awesome.me/webawesome/dist/skills/webawesome-design/` - full-page layout (`<wa-page>`),
  theming with `--wa-*` tokens, brand color, and visual composition.

This file covers only how Web Awesome plugs into the GreyCat/Vite/Lit stack; defer to those skills for
everything Web-Awesome-specific.

## The layout

```
my-project/
|-- project.gcl              # @include("src") - GreyCat backend entrypoint
|-- src/                     # GreyCat backend sources (classic, unchanged)
|-- webroot/                 # build output, served at / (gitignored)
|-- vite.config.ts           # frontend build -> webroot
|-- tsconfig.json
|-- package.json             # vite-plus, lit, @awesome.me/webawesome, @greycat/web
|-- project.d.ts             # generated by `greycat codegen ts` (ambient gc.* types)
`-- app/                     # the entire frontend
    |-- pages/               #   MPA pages - the Vite root; path == URL
    |   |-- index.html + index.ts             ->  /
    |   |-- dashboard/index.html + index.ts   ->  /dashboard/
    |   `-- sensors/index.html + index.ts     ->  /sensors/
    |-- components/          #   reusable Lit components (imported as ~/components/X)
    |-- public/              #   copied as-is into webroot/
    `-- theme.css            #   brand: the --wa-* token overrides
```

- **`~` is an alias for `app/`.** Import shared code as `~/components/x`, never `../../components/x`. It is
  declared in both `vite.config.ts` and `tsconfig.json` so the bundler and the type checker agree.
- **`webroot/` is the public root.** Everything in it is served at the matching URL. Unknown paths return 404 -
  there is no SPA fallback. MPA sidesteps this: every page is a real file, so refresh and deep links resolve.

## Backend vs frontend: what `app/` signals

`src/` always holds the GreyCat backend and is never renamed. The presence of `app/` is what makes a project
fullstack. A backend-only project has no `app/`, no `vite.config.ts`, and no frontend dependencies - it stays a
plain GreyCat project. Never list `app/` in a GreyCat pragma; `@include("app")` would try to load it as `.gcl`.

## Bootstrap

VitePlus ships as a global `vp` CLI plus a local `vite-plus` package. Install `vp` once
(`curl -fsSL https://vite.plus | bash`), then per project:

### `package.json`

```jsonc
{
  "name": "my-project-app",
  "private": true,
  "type": "module",
  "scripts": {
    "build": "vp build"
  },
  "dependencies": {
    "@greycat/web": "https://get.greycat.io/files/sdk/web/dev/8.0/8.0.39-dev.tgz", // example pin - resolve the current one via the recipe below
    "@awesome.me/webawesome": "^3",
    "lit": "^3"
  },
  "devDependencies": {
    "vite-plus": "^0.2"
  }
}
```

`vp install` installs these with the right package manager.

**`@greycat/web` is not on npm.** It ships from GreyCat's registry as a tarball URL that tracks the same
branch and version line as the project's `std` library - branch `dev` or `stable`. Pin the exact tarball in
`package.json`:

```
https://get.greycat.io/files/sdk/web/<branch>/<version>.tgz
```

Resolve the current `<version>` by reading the branch's `latest` marker, then append `.tgz`:

```sh
curl -s https://get.greycat.io/files/sdk/web/dev/latest      # -> 8.0/8.0.39-dev
# dependency: https://get.greycat.io/files/sdk/web/dev/8.0/8.0.39-dev.tgz

curl -s https://get.greycat.io/files/sdk/web/stable/latest   # -> 7.8/7.8.25-stable
# dependency: https://get.greycat.io/files/sdk/web/stable/7.8/7.8.25-stable.tgz
```

Use the branch that matches the project's `std` branch, and re-pin when `std` moves.

### `vite.config.ts`

Explicit config. `greycat dev` serves the API and the built assets from one origin, so nothing needs proxying.
`root: 'app/pages'` makes each page's path its URL; `outDir` targets `webroot/`. The one plugin is
`@greycat/web/vite-plugin` (see below), which handles asset gzipping, `sitemap.xml`, and `robots.txt`.

```ts
import { defineConfig } from 'vite-plus';
import { resolve } from 'node:path';
import { greycat } from '@greycat/web/vite-plugin';

export default defineConfig({
  root: 'app/pages',                 // app/pages/**/index.html -> a page at that URL
  base: './',                         // relative asset URLs so nested pages resolve under webroot/
  appType: 'mpa',                     // no SPA history fallback
  publicDir: resolve('app/public'),   // copied as-is into webroot/
  resolve: { alias: { '~': resolve('app') } },
  plugins: [
    // gzip + (with hostname) sitemap.xml/robots.txt. See below.
    greycat({ /* hostname: 'https://app.example.com' */ }),
  ],
  build: {
    outDir: resolve('webroot'),
    emptyOutDir: true,
    target: 'esnext',
    rollupOptions: {
      // One entry per page. Vite does not auto-discover extra HTML pages.
      input: [
        resolve('app/pages/index.html'),
        resolve('app/pages/dashboard/index.html'),
        resolve('app/pages/sensors/index.html'),
      ],
    },
  },
});
```

A single-page app can omit `rollupOptions.input` entirely - Vite builds `app/pages/index.html` by default. Add
an entry to the list for each new page.

### The `greycat` build plugin

`@greycat/web/vite-plugin` ships in the SDK the app already depends on - no extra package. It runs at build
time only (`apply: 'build'`) and does three things on `writeBundle`, each **on by default and opt-out** via
`false`:

- **`gzip`** - writes a `<file>.gz` next to every asset GreyCat serves pre-compressed (`.js`, `.mjs`, `.cjs`,
  `.css`, `.svg`, `.html`, `.json`, `.wasm`) and **keeps the original**. GreyCat's static server sends the
  `.gz` automatically when the request carries `Accept-Encoding: gzip`, and falls back to the original
  otherwise. Fonts are intentionally excluded: `.woff2` is already compressed and is not in the server's
  gzip whitelist.
- **`sitemap`** - writes `sitemap.xml` from the built HTML pages (one `<loc>` per page; MPA path == URL).
- **`robots`** - writes `robots.txt` (allow-all plus a `Sitemap:` line when `sitemap` is enabled).

**`hostname` gates the SEO files.** `sitemap.xml` and `robots.txt` need an absolute origin, so both are
**skipped when `hostname` is absent** - `greycat()` with no options only gzips. Set `hostname` to the deployed
origin (`https://app.example.com`) to get them. `gzip` never needs `hostname`.

```ts
greycat();                                          // gzip only (no hostname -> no sitemap/robots)
greycat({ hostname: 'https://app.example.com' });   // gzip + sitemap.xml + robots.txt
greycat({ hostname: 'https://app.example.com', sitemap: false }); // gzip + robots.txt, no sitemap
greycat({ gzip: false });                           // disable gzipping
greycat({ gzip: { keepOriginal: false } });         // gzip and delete originals (gzip clients only)
```

### `tsconfig.json`

Mirror the `~` alias and include the generated `project.d.ts` so `gc.*` is typed.

```jsonc
{
  "compilerOptions": {
    "target": "esnext",
    "module": "esnext",
    "moduleResolution": "bundler",
    "lib": ["esnext", "dom", "dom.iterable"],
    "baseUrl": ".",
    "paths": { "~/*": ["app/*"] },
    "strict": true,
    "noEmit": true,
    "skipLibCheck": true,
    "experimentalDecorators": true,  // REQUIRED for Lit: vite-plus (rolldown/oxc) emits unparseable output otherwise
    "useDefineForClassFields": false // REQUIRED for Lit: keep the @property accessor, don't shadow it with a native field
  },
  "include": ["app", "project.d.ts"]
}
```

Both decorator options are load-bearing and their failure is silent - see the first Common pitfall.

### `app/theme.css`

Web Awesome's `webawesome.css` already defines the full `--wa-*` token set and ships light and dark out of the
box. `theme.css` holds only the brand overrides - the handful of `--wa-*` tokens that carry the app's identity -
plus a small page reset. Everything else (surfaces, text, borders, spacing, radii) comes from the Web Awesome
tokens unchanged.

Custom properties are inherited, so every `--wa-*` value declared here on `:root` is visible inside every
component's shadow DOM; components read them with `var(--wa-...)`. Component-specific rules (layout, element
selectors) do **not** belong here - a `.card { ... }` rule in `theme.css` would not cross the shadow boundary.
Those live in the component's own `static styles`.

Dark is the default: every page's `<html>` carries `class="wa-dark"`. Removing it (or setting `wa-light`) flips
to light. A single toggle on `<html>` switches every `wa-*` element and every component together.

```css
/* app/theme.css - brand token overrides + page reset.
   Import AFTER '@awesome.me/webawesome/dist/styles/webawesome.css'. */

:root {
  /* Point Web Awesome's brand color group at the app's brand. The token groups
     (-fill-*, -on-*, -border-*) are documented in the webawesome-design skill;
     these are the high-value ones. */
  --wa-color-brand-fill-loud: #37c43e;
  --wa-color-brand-fill-normal: #4fd24a;
  --wa-color-brand-on-loud: #ffffff;

  --wa-font-family-body: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, Helvetica, Arial, sans-serif;
  --wa-border-radius-m: 4px;
}

body {
  margin: 0;
  color: var(--wa-color-text-normal);
  background: var(--wa-color-surface-default);
  font-family: var(--wa-font-family-body);
  line-height: 1.6;
}
```

### A page

A page is a plain `index.html` that carries the color-scheme class and hosts one Lit root element, plus an
`index.ts` that defines it. The root element owns the page lifecycle - including the init/login gate, since
`gc.sdk.init()` must resolve before any typed call works (see
[Authentication](#authentication-and-the-init-login-flow)). It renders in shadow DOM (the Lit default) and
carries its own `static styles`.

```html
<!-- app/pages/index.html -->
<!doctype html>
<html lang="en" class="wa-dark">
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>My App</title>
    <script type="module" src="./index.ts"></script>
  </head>
  <body>
    <app-home></app-home>
  </body>
</html>
```

```ts
// app/pages/index.ts - the '/' page.
import '@greycat/web/sdk'; // gc runtime: typed calls, init, login (headless - no CSS, no components)
import '@awesome.me/webawesome/dist/styles/webawesome.css'; // --wa-* tokens + light/dark
import '~/theme.css'; // brand overrides (loads AFTER webawesome.css)
import '@awesome.me/webawesome/dist/components/input/input.js'; // <wa-input>
import '@awesome.me/webawesome/dist/components/button/button.js'; // <wa-button>
import { LitElement, html, css } from 'lit';
import { customElement, state } from 'lit/decorators.js';
import '~/components/greeting-panel';

@customElement('app-home')
export class AppHome extends LitElement {
  // Encapsulated to this component's shadow DOM. Reads the global --wa-* tokens
  // (custom properties inherit across the shadow boundary).
  static styles = css`
    form {
      display: grid;
      gap: var(--wa-space-m);
      max-width: 20rem;
    }
    .error {
      color: var(--wa-color-danger-on-quiet);
    }
  `;

  @state() private ready = false;
  @state() private error = '';
  @state() private greeting = '';

  async connectedCallback() {
    super.connectedCallback();
    // init() loads the ABI over an authenticated endpoint. It succeeds if a
    // session cookie from a prior login exists; otherwise it throws and we show
    // the login form. gc.<module>.* only works after it resolves.
    try {
      await gc.sdk.init();
      await this.load();
    } catch {
      this.ready = false;
    }
  }

  private async submit(e: SubmitEvent) {
    e.preventDefault();
    const data = new FormData(e.target as HTMLFormElement);
    const username = data.get('username') as string;
    const password = data.get('password') as string;
    try {
      await gc.sdk.init({ auth: { username, password } });
      await this.load();
    } catch {
      this.error = 'Invalid credentials';
    }
  }

  private async load() {
    this.greeting = await gc.project.hello('world'); // typed via project.d.ts
    this.ready = true;
  }

  render() {
    if (!this.ready) {
      return html`
        <form @submit=${this.submit}>
          <wa-input name="username" label="Username" required></wa-input>
          <wa-input name="password" type="password" label="Password" required></wa-input>
          <wa-button type="submit" variant="brand">Sign in</wa-button>
          ${this.error ? html`<small class="error">${this.error}</small>` : ''}
        </form>
      `;
    }
    return html`<greeting-panel name=${this.greeting}></greeting-panel>`;
  }
}
```

### A reusable component

Reach for a component in `app/components/` only when a view is reused across pages. It renders in shadow DOM
(the Lit default), owns its layout in `static styles`, and imports the `wa-*` elements it renders so it is
self-contained. Style with the global `--wa-*` tokens, which inherit across the shadow boundary.

```ts
// app/components/greeting-panel.ts
import { LitElement, html, css } from 'lit';
import { customElement, property } from 'lit/decorators.js';
import '@awesome.me/webawesome/dist/components/button/button.js';

@customElement('greeting-panel')
export class GreetingPanel extends LitElement {
  static styles = css`
    section {
      padding: var(--wa-space-l);
    }
  `;

  @property() name = '';

  render() {
    return html`
      <section>
        <p>Hello, ${this.name}</p>
        <wa-button variant="brand">Refresh</wa-button>
      </section>
    `;
  }
}
```

## The UI kit: Web Awesome

Every visual element is a Web Awesome `wa-*` custom element. Import `webawesome.css` once per page for the
tokens and light/dark, then cherry-pick each component's JS module where you use it - the import registers the
element and bundles only what you use.

```ts
import '@awesome.me/webawesome/dist/styles/webawesome.css';      // tokens + light/dark (once per page)
import '@awesome.me/webawesome/dist/components/button/button.js'; // <wa-button>
import '@awesome.me/webawesome/dist/components/dialog/dialog.js'; // <wa-dialog>
```

- **Custom elements never self-close.** Write `<wa-input></wa-input>`, not `<wa-input />`.
- **Form-control events use native names, not a `wa-` prefix.** `wa-input`, `wa-checkbox`, `wa-select` and the
  rest emit `input` and `change`; only a few component-specific events keep the prefix (`wa-clear`,
  `wa-invalid`). Bind the Lit handler to the exact event name in that component's `webawesome` skill reference
  (`node_modules/@awesome.me/webawesome/dist/skills/webawesome/references/components/<name>.md`). A wrong name
  like `@wa-change` on a checkbox binds nothing and fails silently - the click does nothing and the build stays
  green. Slots and CSS parts follow Web Awesome's own conventions - see the `webawesome` skill.
- **Full-page shells use `<wa-page>`.** For app-shell/sidebar/header layout, follow the `webawesome-design`
  skill's `<wa-page>` recipe rather than hand-rolling a grid. `<wa-page>` renders in light DOM at the page
  level (outside any Lit shadow root).
- **Theming.** Every component reads `--wa-*` tokens; `app/theme.css` re-brands them by overriding a few. Dark
  is the default via the `wa-dark` class on `<html>`; removing it (or `wa-light`) flips to light.

**Assets (icons).** Web Awesome auto-detects its asset base path from the loaded script. If icons or images
fail to resolve under the bundler, set the base path explicitly once at startup:

```ts
import { setBasePath } from '@awesome.me/webawesome/dist/webawesome.js';
setBasePath('/webawesome'); // a path under webroot/ where you copied dist/assets
```

## Type bindings: `greycat codegen ts`

`greycat codegen ts` reads the compiled program and writes `project.d.ts` - an ambient declaration that extends
the SDK's global `gc` namespace with every public type and `@expose`d function. It emits no imports; the
`tsconfig.json` `include` above is what puts `gc.project.hello(...)` in scope with full types.

The segment after `gc.` is the GreyCat **module**: the source file's basename, the same `<module>` that forms
the path-RPC URL `/<module>::<fn>`. A function in `src/tasks.gcl` is `gc.tasks.foo(...)` (URL `/tasks::foo`);
one written in `project.gcl` itself is `gc.project.foo(...)`. The examples here live in `project.gcl`.

```sh
greycat codegen ts
```

Regenerate it whenever the backend ABI changes (new/changed `@expose` function or type). A client built against
a stale ABI gets HTTP 422 at runtime.

`project.d.ts` types the calls at compile time, but the runtime `gc.<module>.*` objects (typed call functions,
enum statics like `gc.project.Status.active`) are built from the ABI **during `gc.sdk.init()`**. Do not touch
`gc.<module>.*` at import/module-eval time - only after init resolves.

## Calling the backend through the SDK

`import '@greycat/web/sdk'` loads the SDK runtime: the `gc` global, `init`, `login`, and the typed call
bindings. It is headless - it registers no components and injects no CSS. Import it once.

```ts
import '@greycat/web/sdk';

await gc.sdk.init(); // registers the default instance as gc.$.default
const total = await gc.project.add(10, 32); // typed call -> 42
const task = await gc.project.import_data.spawn(url); // run server-side as a background task
```

Every generated call takes the function's args followed by two optional params: `$g` (which instance) and
`$signal` (an `AbortSignal`). The untyped escape hatch is `gc.$.default.call<T>('module::fn', args, signal)`.

Return values are real SDK objects, not plain JSON: a returned `time`/`duration`/`geo` carries the same helpers
as its GCL counterpart (`time.format(...)`, `.toInputValue()`, `.ord(...)`) - see [stdlib.md](stdlib.md).
Don't hand-roll `Date` math. For the exact TypeScript signatures of any native type, read the generated SDK
source under `node_modules/@greycat/web/src/sdk/std_n/core`.

A GCL `enum` becomes a class exposing one static per entry, each instance carrying a `key` string. Pass the
static to a call; read `.key` for the label. Over the wire the entry serializes as its bare name.

```ts
// GCL: enum Status { active; archived }
await gc.project.set_status(id, gc.project.Status.active); // pass the enum instance
const s = await gc.project.status_of(id); // -> gc.project.Status instance
label.textContent = s.key; // "active"
```

### Authentication and the init/login flow

Auth is the default. A request with no session reaches only functions marked `@permission("public")`; every
other `@expose` returns 401. `gc.sdk.init()` itself loads the ABI over an authenticated endpoint, so it also
needs a session. That dictates the ordering the page's root element uses:

1. Call `gc.sdk.init()` with no args. It succeeds if a session cookie from a prior login is present.
2. If it throws, render a login form and call `gc.sdk.init({ auth: { username, password } })` (or `{ token }`),
   which authenticates **and** loads the ABI.
3. Only once init has resolved are `gc.<module>.*` calls usable.

```ts
await gc.sdk.init({ auth: { username, password } }); // or { token }
```

The resolved token rides the `Authorization` header on every subsequent call. Do **not** add
`@permission("public")` to endpoints just to skip login during development - that exposes them to every
anonymous caller. It is reserved for endpoints that genuinely must serve anonymous traffic (`login`, a health
probe) and is a deliberate project-owner decision.

The raw JSON-RPC and path-RPC HTTP endpoints still exist (see [runtime.md](runtime.md)); the SDK calls them for
you, and app code should go through the SDK.

## Running and building

```sh
vp install            # install frontend deps
greycat install       # download lib/std and bin/greycat
greycat codegen ts    # generate project.d.ts
greycat dev           # build watcher (vp build --watch) + serve API and assets on one origin
```

`greycat dev` detects `vite.config.ts` in the project root and spawns the watcher automatically; the API and the
built `webroot/` are served together on port 8080. It accepts every `greycat serve` option (`--port`, `--log`,
...). See [cli.md](cli.md).

For production, build both sides:

```sh
vp build              # populates webroot/
greycat build         # produces project.gcp
```

Deploy artifacts: `project.gcp` (or `project.gcl` + `src/` + populated `lib/`), `bin/greycat`, the built
`webroot/`, and a writable `gcdata/` and `files/`.

Suggested `.gitignore`:

```
node_modules/
webroot/
project.d.ts
```

## Common pitfalls

1. **Omitting `experimentalDecorators: true`.** vite-plus (rolldown/oxc) does not transform standard
   decorators: it passes `@customElement` through as unparseable syntax and silently drops `@property`. `vp
   build` still reports success, and the page loads blank with a `SyntaxError`. Set both decorator options in
   `tsconfig.json` exactly as prescribed (`experimentalDecorators: true`, `useDefineForClassFields: false`).
2. **Touching `gc.<module>.*` before init.** The runtime call bindings are built during `gc.sdk.init()`.
   Referencing `gc.project.*` before init resolves throws. Gate on init in the page's root element.
3. **Forgetting `greycat codegen ts`.** Without `project.d.ts`, `gc.project.*` is untyped or missing.
   Regenerate after every ABI change; a stale client gets HTTP 422.
4. **Self-closing a `wa-*` tag.** Custom elements cannot self-close. `<wa-input />` silently swallows the rest
   of the markup; write `<wa-input></wa-input>`.
5. **`theme.css` imported before `webawesome.css`.** The brand overrides in `theme.css` win by source order;
   load `webawesome.css` first or the overrides lose.
6. **Putting component styles in `theme.css`.** Only design tokens (custom properties) and the page reset
   belong in `theme.css`; those inherit into shadow DOM. A regular element/class rule there (e.g. `.card {
   padding: ... }`) will not cross the shadow boundary. Put component layout and element styles in that
   component's `static styles` block, referencing the global `--wa-*` tokens.
7. **Importing anything but the SDK from `@greycat/web`.** Use `@greycat/web/sdk` only. It is headless; all UI
   comes from Web Awesome. There is no `@greycat/web` component set or CSS to import.
8. **Relative import paths.** Use the `~` alias (`~/components/x`), declared in both `vite.config.ts` and
   `tsconfig.json`. Drifting one from the other breaks either the build or the type checker.
9. **Expecting an SPA fallback.** Unknown paths return 404. Every navigable URL must be a real
   `app/pages/**/index.html`. This is why the architecture is MPA.
10. **Missing page entries.** Vite does not auto-discover extra HTML pages; add each page's `index.html` to
    `build.rollupOptions.input`.
11. **`app/` in a GreyCat pragma.** `app/` is the frontend, not GreyCat source. Never `@include("app")`.
12. **Hardcoded colors or sizes.** Every value comes from a `--wa-*` token (re-branded in `theme.css`). A raw
    hex in a component is a mistake.
13. **Binding a Web Awesome event under the wrong name.** Form controls emit native-named events (`input`,
    `change`), not `wa-`-prefixed ones. `@wa-change` on a `wa-checkbox` binds nothing, the control looks
    interactive but does nothing, and both builds pass. Confirm the event name in the component's `webawesome`
    skill reference, and let a Playwright spec catch the dead handler.
14. **Installing frontend deps with npm/npx or bare pnpm.** `vp install` uses the pinned pnpm; `pnpm add`
    directly relinks the store across pnpm majors and errors, and `npx` trips the `devEngines` pin. Add deps to
    `package.json`, run `vp install`, and invoke tools through `pnpm exec` / `pnpm run`.
15. **Assuming a fresh database per test.** `gcdata/` is one persistent store and `--store` only caps its size.
    Playwright specs must create uniquely-labeled data and clean it up, not assume an empty list.
