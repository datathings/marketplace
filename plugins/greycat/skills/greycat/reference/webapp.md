# Adding a webapp to a GreyCat project

The one prescribed way to ship a web frontend with a GreyCat backend. It is opinionated on purpose: every
GreyCat webapp uses the same toolchain, layout, and design tokens, so any of them reads as "a classic GreyCat
app." Deviate only when a project has a concrete reason to.

The stack:

- **VitePlus** (`vp`) as the toolchain, building `app/` into `webroot/`.
- **MPA** (multi-page): each route is a real HTML page; the URL is the file path. No SPA router.
- **Lit** in **light DOM**: one root element per route, plus a component for any view reused across routes.
- **UI kit**: Shoelace (`sl-*`) for atomic controls, `@greycat/web` (`gui-*`) for rich and GreyCat-aware
  widgets. Both are themed by `greycat.css`, which is a Shoelace theme.
- **`app/theme.css`** as the single design-token file: the app's own `--app-*` variables, plus a small layer
  that re-skins Shoelace/GreyCat by overriding their `--sl-*` tokens.
- **`@greycat/web` SDK** for every backend call, which makes **`greycat codegen ts`** mandatory.

## The layout

```
my-project/
├── project.gcl              # @include("src") - GreyCat backend entrypoint
├── src/                     # GreyCat backend sources (classic, unchanged)
├── webroot/                 # build output, served at / (gitignored)
├── vite.config.ts           # frontend build -> webroot
├── tsconfig.json
├── package.json             # vite-plus, lit, @shoelace-style/shoelace, @greycat/web
├── project.d.ts             # generated by `greycat codegen ts` (ambient gc.* types)
└── app/                     # the entire frontend
    ├── routes/              #   MPA pages - the Vite root; path == URL
    │   ├── index.html + index.ts            ->  /
    │   ├── dashboard/index.html + index.ts   ->  /dashboard/
    │   └── sensors/index.html + index.ts     ->  /sensors/
    ├── components/          #   reusable light-DOM Lit views (imported as ~/components/X)
    ├── public/              #   copied as-is into webroot/
    └── theme.css            #   the --app-* design tokens + the sl-* re-skin
```

- **`~` is an alias for `app/`.** Import shared code as `~/components/x`, never `../../components/x`. It is
  declared in both `vite.config.ts` and `tsconfig.json` so the bundler and the type checker agree.
- **`webroot/` is the public root.** Everything in it is served at the matching URL. Unknown paths return 404 -
  there is no SPA fallback. MPA sidesteps this: every route is a real file, so refresh and deep links resolve.

## Backend vs frontend: what `app/` signals

`src/` always holds the GreyCat backend and is never renamed. The presence of `app/` is what makes a project
fullstack. A backend-only project has no `app/`, no `vite.config.ts`, and no frontend dependencies - it stays a
plain GreyCat project. Never list `app/` in a GreyCat pragma; `@include("app")` would try to load it as `.gcl`.

## Bootstrap

VitePlus ships as a global `vp` CLI plus a local `vite-plus` package. Install `vp` once
(`curl -fsSL https://vite.plus | bash`), then per project:

### `package.json`

```jsonc
{
  "name": "my-project-app",
  "private": true,
  "type": "module",
  "scripts": {
    "build": "vp build"
  },
  "dependencies": {
    "@greycat/web": "https://get.greycat.io/files/sdk/web/dev/8.0/8.0.39-dev.tgz", // example pin - resolve the current one via the recipe below
    "@shoelace-style/shoelace": "^2.20.1", // must satisfy @greycat/web's shoelace peer range
    "lit": "^3"
  },
  "devDependencies": {
    "vite-plus": "^0.2"
  }
}
```

`vp install` installs these with the right package manager. `@greycat/web` depends on Shoelace as a peer, so
the app installs `@shoelace-style/shoelace` directly; Shoelace is Lit-based and shares the `lit` install.

**`@greycat/web` is not on npm.** It ships from GreyCat's registry as a tarball URL that tracks the same
branch and version line as the project's `std` library - branch `dev` or `stable`. Pin the exact tarball in
`package.json`:

```
https://get.greycat.io/files/sdk/web/<branch>/<version>.tgz
```

Resolve the current `<version>` by reading the branch's `latest` marker, then append `.tgz`:

```sh
curl -s https://get.greycat.io/files/sdk/web/dev/latest      # -> 8.0/8.0.39-dev
# dependency: https://get.greycat.io/files/sdk/web/dev/8.0/8.0.39-dev.tgz

curl -s https://get.greycat.io/files/sdk/web/stable/latest   # -> 7.8/7.8.25-stable
# dependency: https://get.greycat.io/files/sdk/web/stable/7.8/7.8.25-stable.tgz
```

Use the branch that matches the project's `std` branch, and re-pin when `std` moves.

### `vite.config.ts`

Explicit config - no plugin. `greycat dev` serves the API and the built assets from one origin, so nothing
needs proxying. `root: 'app/routes'` makes each page's path its URL; `outDir` targets `webroot/`.

```ts
import { defineConfig } from 'vite-plus';
import { resolve } from 'node:path';

export default defineConfig({
  root: 'app/routes',                 // app/routes/**/index.html -> a page at that URL
  base: './',                         // relative asset URLs so nested pages resolve under webroot/
  appType: 'mpa',                     // no SPA history fallback
  publicDir: resolve('app/public'),   // copied as-is into webroot/
  resolve: { alias: { '~': resolve('app') } },
  build: {
    outDir: resolve('webroot'),
    emptyOutDir: true,
    target: 'esnext',
    rollupOptions: {
      // One entry per route. Vite does not auto-discover extra HTML pages.
      input: [
        resolve('app/routes/index.html'),
        resolve('app/routes/dashboard/index.html'),
        resolve('app/routes/sensors/index.html'),
      ],
    },
  },
});
```

A single-page app can omit `rollupOptions.input` entirely - Vite builds `app/routes/index.html` by default. Add
an entry to the list for each new route.

### `tsconfig.json`

Mirror the `~` alias and include the generated `project.d.ts` so `gc.*` is typed.

```jsonc
{
  "compilerOptions": {
    "target": "esnext",
    "module": "esnext",
    "moduleResolution": "bundler",
    "lib": ["esnext", "dom", "dom.iterable"],
    "baseUrl": ".",
    "paths": { "~/*": ["app/*"] },
    "strict": true,
    "noEmit": true,
    "skipLibCheck": true,
    "experimentalDecorators": true,  // REQUIRED for Lit: vite-plus (rolldown/oxc) emits unparseable output otherwise
    "useDefineForClassFields": false // REQUIRED for Lit: keep the @property accessor, don't shadow it with a native field
  },
  "include": ["app", "project.d.ts"]
}
```

Both decorator options are load-bearing and their failure is silent - see the first Common pitfall.

### `app/theme.css`

One file owns the design. It has two parts: the app's own `--app-*` tokens (the source of truth for the app's
markup and components), and a re-skin layer that overrides Shoelace/GreyCat tokens to consume those `--app-*`
values, so every `sl-*`/`gui-*` element adopts the brand. Import it **after** `greycat.css` so the re-skin wins
the cascade.

Dark is the default, matching `greycat.css`. Light is applied by putting the class `sl-theme-light` on
`<html>` - the same class that flips `greycat.css` - so a single toggle switches the app tokens and every

### A route page

A route is a plain `index.html` that carries the theme class and hosts one light-DOM Lit root element, plus an
`index.ts` that defines it. The root element owns the page lifecycle - including the init/login gate, since
`gc.sdk.init()` must resolve before any typed call or `gui-*` element works (see
[Authentication](#authentication-and-the-init-login-flow)).

```html
<!-- app/routes/index.html -->
<!doctype html>
<html lang="en" class="sl-theme-dark">
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>My App</title>
    <script type="module" src="./index.ts"></script>
  </head>
  <body>
    <app-home></app-home>
  </body>
</html>
```

```ts
// app/routes/index.ts - the '/' page.
import '@greycat/web/sdk'; // gc runtime: typed calls, init, login
import '@greycat/web/greycat.css'; // themes every sl-*/gui-* element
import '~/theme.css'; // app tokens + re-skin (loads AFTER greycat.css)
import '@shoelace-style/shoelace/dist/components/input/input.js'; // <sl-input>
import '@shoelace-style/shoelace/dist/components/button/button.js'; // <sl-button>
import { LitElement, html } from 'lit';
import { customElement, state } from 'lit/decorators.js';
import '~/components/greeting-panel';

@customElement('app-home')
export class AppHome extends LitElement {
  createRenderRoot() {
    return this; // light DOM: theme.css and the components cascade in
  }

  @state() private ready = false;
  @state() private error = '';
  @state() private greeting = '';

  async connectedCallback() {
    super.connectedCallback();
    // init() loads the ABI over an authenticated endpoint. It succeeds if a
    // session cookie from a prior login exists; otherwise it throws and we show
    // the login form. gc.<module>.* and gui-* tags only work after it resolves.
    try {
      await gc.sdk.init();
      await this.load();
    } catch {
      this.ready = false;
    }
  }

  private async submit(e: SubmitEvent) {
    e.preventDefault();
    const data = new FormData(e.target as HTMLFormElement);
    try {
      await gc.sdk.init({
        auth: { username: String(data.get('username')), password: String(data.get('password')) },
      });
      await this.load();
    } catch {
      this.error = 'Invalid credentials';
    }
  }

  private async load() {
    this.greeting = await gc.project.hello('world'); // typed via project.d.ts
    this.ready = true;
  }

  render() {
    if (!this.ready) {
      return html`
        <form @submit=${this.submit} style="display:grid; gap:var(--app-space-3); max-width:20rem">
          <sl-input name="username" label="Username" required></sl-input>
          <sl-input name="password" type="password" label="Password" required></sl-input>
          <sl-button type="submit" variant="primary">Sign in</sl-button>
          ${this.error ? html`<small style="color:var(--app-danger)">${this.error}</small>` : ''}
        </form>
      `;
    }
    return html`<greeting-panel name=${this.greeting}></greeting-panel>`;
  }
}
```

### A reusable component

Reach for a component in `app/components/` only when a view is reused across routes. Render into **light DOM**
so `theme.css` and the component styles cascade in, and import the `sl-*`/`gui-*` elements it renders so it is
self-contained.

```ts
// app/components/greeting-panel.ts
import { LitElement, html } from 'lit';
import { customElement, property } from 'lit/decorators.js';
import '@shoelace-style/shoelace/dist/components/button/button.js';

@customElement('greeting-panel')
export class GreetingPanel extends LitElement {
  @property() name = '';

  createRenderRoot() {
    return this; // light DOM
  }

  render() {
    return html`
      <section style="padding: var(--app-space-4)">
        <p>Hello, ${this.name}</p>
        <sl-button variant="primary">Refresh</sl-button>
      </section>
    `;
  }
}
```

## The UI kit: Shoelace atomics + GreyCat widgets

One component family, one theme. `greycat.css` is a Shoelace theme, so it styles both tiers uniformly and
supplies light/dark for free. Do **not** import Shoelace's own `themes/light.css` or `dark.css`; `greycat.css`
replaces them.

- **Atomic controls** come from Shoelace (`sl-*`): button, input, checkbox, switch, select, dialog, tab group,
  tooltip, and the rest. `@greycat/web` does not ship these; import each one you use:

  ```ts
  import '@shoelace-style/shoelace/dist/components/button/button.js'; // <sl-button>
  ```

- **Rich / GreyCat-aware widgets** come from `@greycat/web` (`gui-*`): tables, charts, maps, the type-driven
  form generator (`gui-object` + `gui-input-*`), sign-in, task views. Import each one you use, or the bundle:

  ```ts
  import '@greycat/web/components/table'; // <gui-table>
  import '@greycat/web/components/all.js'; // or: every gui-* at once
  ```

**Theming.** Both tiers read Shoelace's `--sl-*` tokens. `greycat.css` is dark by default; switch to light by
putting `sl-theme-light` on `<html>`. `app/theme.css` re-skins both by overriding a few `--sl-*` tokens.

**Deferred registration.** `gui-*` tags are not defined until `gc.sdk.init()` resolves, so render them only
after init. The route's init gate above already handles this.

**Icons.** Shoelace's built-in UI icons (chevrons, checks) work out of the box. To use `<sl-icon>` with the
default icon library, copy Shoelace's `dist/assets/icons` into `app/public/` and point Shoelace at it:

```ts
import { setBasePath } from '@shoelace-style/shoelace/dist/utilities/base-path.js';
setBasePath('/assets'); // wherever you copied dist/assets
```

## Type bindings: `greycat codegen ts`

`greycat codegen ts` reads the compiled program and writes `project.d.ts` - an ambient declaration that extends
the SDK's global `gc` namespace with every public type and `@expose`d function. It emits no imports; the
`tsconfig.json` `include` above is what puts `gc.project.hello(...)` in scope with full types.

The segment after `gc.` is the GreyCat **module**: the source file's basename, the same `<module>` that forms
the path-RPC URL `/<module>::<fn>`. A function in `src/tasks.gcl` is `gc.tasks.foo(...)` (URL `/tasks::foo`);
one written in `project.gcl` itself is `gc.project.foo(...)`. The examples here live in `project.gcl`.

```sh
greycat codegen ts
```

Regenerate it whenever the backend ABI changes (new/changed `@expose` function or type). A client built against
a stale ABI gets HTTP 422 at runtime.

`project.d.ts` types the calls at compile time, but the runtime `gc.<module>.*` objects (typed call functions,
enum statics like `gc.project.Status.active`) are built from the ABI **during `gc.sdk.init()`**. Do not touch
`gc.<module>.*` at import/module-eval time - only after init resolves.

## Calling the backend through the SDK

`import '@greycat/web/sdk'` loads the SDK runtime: the `gc` global, `init`, `login`, and the typed call
bindings. Import it once; import the `gui-*` components you use individually rather than the umbrella
`@greycat/web`, which registers every component.

```ts
import '@greycat/web/sdk';

await gc.sdk.init(); // registers the default instance as gc.$.default
const total = await gc.project.add(10, 32); // typed call -> 42
const task = await gc.project.import_data.spawn(url); // run server-side as a background task
```

Every generated call takes the function's args followed by two optional params: `$g` (which instance) and
`$signal` (an `AbortSignal`). The untyped escape hatch is `gc.$.default.call<T>('module::fn', args, signal)`.

Return values are real SDK objects, not plain JSON: a returned `time`/`duration`/`geo` carries the same helpers
as its GCL counterpart (`time.format(...)`, `.toInputValue()`, `.ord(...)`) - see [stdlib.md](stdlib.md).
Don't hand-roll `Date` math. For the exact TypeScript signatures of any native type, read the generated SDK
source under `node_modules/@greycat/web/src/sdk/std_n/core`.

A GCL `enum` becomes a class exposing one static per entry, each instance carrying a `key` string. Pass the
static to a call; read `.key` for the label. Over the wire the entry serializes as its bare name.

```ts
// GCL: enum Status { active; archived }
await gc.project.set_status(id, gc.project.Status.active); // pass the enum instance
const s = await gc.project.status_of(id); // -> gc.project.Status instance
label.textContent = s.key; // "active"
```

### Authentication and the init/login flow

Auth is the default. A request with no session reaches only functions marked `@permission("public")`; every
other `@expose` returns 401. `gc.sdk.init()` itself loads the ABI over an authenticated endpoint, so it also
needs a session. That dictates the ordering the route's root element uses:

1. Call `gc.sdk.init()` with no args. It succeeds if a session cookie from a prior login is present.
2. If it throws, render a login form and call `gc.sdk.init({ auth: { username, password } })` (or `{ token }`),
   which authenticates **and** loads the ABI.
3. Only once init has resolved are `gc.<module>.*` calls and `gui-*` elements usable.

```ts
await gc.sdk.init({ auth: { username, password } }); // or { token }
```

The resolved token rides the `Authorization` header on every subsequent call. Do **not** add
`@permission("public")` to endpoints just to skip login during development - that exposes them to every
anonymous caller. It is reserved for endpoints that genuinely must serve anonymous traffic (`login`, a health
probe) and is a deliberate project-owner decision.

The raw JSON-RPC and path-RPC HTTP endpoints still exist (see [runtime.md](runtime.md)); the SDK calls them for
you, and app code should go through the SDK.

## Running and building

```sh
vp install            # install frontend deps
greycat install       # download lib/std and bin/greycat
greycat codegen ts    # generate project.d.ts
greycat dev           # build watcher (vp build --watch) + serve API and assets on one origin
```

`greycat dev` detects `vite.config.ts` in the project root and spawns the watcher automatically; the API and the
built `webroot/` are served together on port 8080. It accepts every `greycat serve` option (`--port`, `--log`,
...). See [cli.md](cli.md).

For production, build both sides:

```sh
vp build              # populates webroot/
greycat build         # produces project.gcp
```

Deploy artifacts: `project.gcp` (or `project.gcl` + `src/` + populated `lib/`), `bin/greycat`, the built
`webroot/`, and a writable `gcdata/` and `files/`.

Suggested `.gitignore`:

```
node_modules/
webroot/
project.d.ts
```

## Common pitfalls

1. **Omitting `experimentalDecorators: true`.** vite-plus (rolldown/oxc) does not transform standard
   decorators: it passes `@customElement` through as unparseable syntax and silently drops `@property`. `vp
   build` still reports success, and the page loads blank with a `SyntaxError`. Set both decorator options in
   `tsconfig.json` exactly as prescribed (`experimentalDecorators: true`, `useDefineForClassFields: false`).
2. **Touching `gc.<module>.*` or `gui-*` before init.** The runtime call bindings and the `gui-*` element
   definitions are built during `gc.sdk.init()`. Referencing `gc.project.*` or rendering a `gui-*` tag before
   init resolves throws or renders nothing. Gate on init in the route's root element.
3. **Forgetting `greycat codegen ts`.** Without `project.d.ts`, `gc.project.*` is untyped or missing.
   Regenerate after every ABI change; a stale client gets HTTP 422.
4. **Importing Shoelace's own theme.** `greycat.css` is the theme. Importing `themes/light.css` or `dark.css`
   alongside it double-defines the `--sl-*` tokens and fights the re-skin. Import only `greycat.css`.
5. **`theme.css` imported before `greycat.css`.** The re-skin in `theme.css` overrides `--sl-*` tokens by
   source order; load `greycat.css` first or the overrides lose.
6. **Shadow-DOM components in the app.** App components must render to light DOM (`createRenderRoot() { return
   this; }`) or `theme.css` will not reach inside them. Shadow encapsulation is for distributable libraries, not
   app views.
7. **Relative import paths.** Use the `~` alias (`~/components/x`), declared in both `vite.config.ts` and
   `tsconfig.json`. Drifting one from the other breaks either the build or the type checker.
8. **Expecting an SPA fallback.** Unknown paths return 404. Every navigable URL must be a real
   `app/routes/**/index.html`. This is why the architecture is MPA.
9. **Missing route entries.** Vite does not auto-discover extra HTML pages; add each route's `index.html` to
   `build.rollupOptions.input`.
10. **`app/` in a GreyCat pragma.** `app/` is the frontend, not GreyCat source. Never `@include("app")`.
11. **Hardcoded colors or sizes.** Every value comes from a `--app-*` token in `theme.css`. A raw hex in a
    component is a mistake.
