# @greycat/web — Development Guide

## Mental Model

`@greycat/web` is **not** React, Vue, or Svelte. Key differences:

- **Web Components** (Shadow DOM + Custom Elements) are the component model. All `gui-*` and `sl-*` elements are standard Custom Elements — usable from plain HTML, vanilla JS, or JSX.
- **JSX is optional.** It's a thin convenience layer over `document.createElement()`. Everything JSX does, you can do with the DOM API directly. Use whichever style fits the situation.
- There is **no reactivity system**. To update the UI, hold a reference and set properties imperatively.
- **Shoelace** (`sl-*`) is the design system — all components are pre-registered and fully typed.
- **GreyCat backend** provides typed data via RPC. `gc.sdk.init()` loads the ABI (type metadata) and registers all `gui-*` components.
- Apps are **Multi-Page Applications** (MPA). Each page is a separate HTML entry point with its own script.

## Project Setup

Scaffold a new project with VitePlus+:

```bash
vp create github:datathings/greycat-web-template
cd your-app-name
pnpm run setup
pnpm install
```

`pnpm run setup` fetches the latest GreyCat core and `@greycat/web` versions, updating `project.gcl` and `package.json` automatically.

### What the Template Provides

The scaffolded project uses the `greycat()` VitePlus+ plugin (`vite.config.ts`) which auto-configures:
- `base: './'` — relative URLs
- `appType: 'mpa'` — multi-page app
- `root: 'app'` — VitePlus+ root directory
- `resolve.alias: { '~': 'app' }` — tilde imports
- Proxies `POST` (RPC) and `/files/*` (GET/PUT/DELETE) to GreyCat backend at `http://127.0.0.1:8080`
- Auto-discovers all `.html` files under `app/` as entry points

Plugin options: `greycat({ greycat: 'http://custom:9090', gzip: true, noDefaultConfig: false })`

### Key Configuration Details

If you need to understand or customize the generated config:

- **`tsconfig.json`**: Uses `"jsx": "react-jsx"` with `"jsxImportSource": "@greycat/web"` for JSX support.
- **`vite.config.ts`**: Imports `defineConfig` from `"vite-plus"` and `greycat` from `@greycat/web/vite-plugin`.
- **`project.d.ts`**: Auto-generated types for `gc.project.*` — never edit this file. Generated by `greycat codegen ts`.

## Page Entry Pattern

### HTML Shell

Every page is a minimal HTML file that loads a `.ts` or `.tsx` script:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Dashboard</title>
  </head>
  <body>
    <script type="module" src="./index.tsx"></script>
  </body>
</html>
```

### Entry Script (with JSX)

```tsx
import '@greycat/web';
import '@greycat/web/greycat.css';

await gc.sdk.init({ debug: import.meta.env.DEV });

const data = await gc.project.myFunction();

document.body.appendChild(
  <gui-layout>
    <div slot="header">My App</div>
    <gui-nav slot="navigation" />
    <div slot="main">
      <gui-table value={data} globalFilter />
    </div>
  </gui-layout>,
);
```

### Entry Script (vanilla JS — no JSX)

The same page without JSX, using the DOM API directly:

```ts
import '@greycat/web';
import '@greycat/web/greycat.css';

await gc.sdk.init({ debug: import.meta.env.DEV });

const data = await gc.project.myFunction();

const header = document.createElement('div');
header.slot = 'header';
header.textContent = 'My App';

const nav = document.createElement('gui-nav');
nav.slot = 'navigation';

const table = document.createElement('gui-table');
table.value = data;
table.globalFilter = true;

const main = document.createElement('div');
main.slot = 'main';
main.appendChild(table);

const layout = document.createElement('gui-layout');
layout.append(header, nav, main);
document.body.appendChild(layout);
```

Both styles are equivalent. JSX is syntactic sugar — pick whichever is clearer for the situation. You can mix them freely in the same file (`.tsx`).

### Key Points

- `import '@greycat/web'` **must** come first — it registers all `gui-*` and `sl-*` components.
- `gc.sdk.init()` connects to the backend, loads the ABI, and must complete before using any `gui-*` component or `gc.project.*` call.
- Top-level `await` works because `<script type="module">`.
- Use `greycat.css` for core styles or `greycat-full.css` for classless HTML element styling.
- CSS modules are supported: `import s from './index.module.css'` then `className={s.myClass}`.

## JSX Runtime (Optional)

JSX is a convenience layer, not a requirement. This section covers its behavior for when you use it.

### Return Type Is a Real DOM Node

```tsx
const div = <div className="foo">Hello</div>;         // HTMLDivElement
const frag = <>item 1, item 2</>;                      // DocumentFragment
const table = (<gui-table value={data} />) as GuiTable; // cast for typed access
```

JSX returns `Node`. Cast with `as` when you need typed access to component-specific properties or methods.

### Props Mapping

| Prop | Behavior |
|---|---|
| `className` | `string` → `classList.add()`; `string[]` → adds all; `{ name: boolean }` → conditional add/remove |
| `style` | `string` → sets `cssText`; object → assigns each property, `--*` keys use `setProperty()` |
| `$ref` | Callback `(el) => void` invoked at creation (element may not be in DOM yet) |
| `slot` | Sets `element.slot` |
| `part` | Adds to `element.part` |
| `exportparts` | Sets `exportparts` attribute |
| `on*` | `addEventListener(name.slice(2), handler)` — e.g. `onclick`, `ongui-change`, `onsl-show` |
| `attr:*` | `setAttribute(name.slice(5), value)` — for HTML attributes without a DOM property |
| Everything else | Direct property assignment: `element[key] = value` (if key exists on element) |
| `null` / `undefined` | Silently skipped |

Examples:

```tsx
// Conditional classes
<div className={{ active: isActive, hidden: !visible }}>...</div>

// Style with CSS variables
<div style={{ display: 'flex', gap: '8px', '--my-color': 'red' }}>...</div>

// Callback ref
<gui-chart $ref={(el) => { chartRef = el; }} />

// HTML attribute (no DOM property)
<div attr:data-id="123" attr:role="button" />

// Event with hyphenated name
<gui-table ongui-change={(ev) => console.log(ev.detail)} />
```

### The `setAttrs` Optimization

When JSX creates a `gui-*` component that has a `setAttrs()` method, the runtime calls `setAttrs(props)` **once** instead of setting each property individually. This lets components batch their internal updates. Event handlers (`on*`) and `$ref` are still applied separately.

When setting multiple properties imperatively, you can use `setAttrs` to batch updates and avoid multiple renders:
```ts
table.setAttrs({ value: data, globalFilter: true, columns: cols });
```

### `cx()` Utility

```ts
import { cx } from '@greycat/web/jsx-runtime';

cx(element, 'foo');                          // add single class
cx(element, ['foo', 'bar']);                 // add multiple
cx(element, { active: true, hidden: false }); // conditional add/remove
```

### Children

- Arrays, `NodeList`, `HTMLCollection` → flattened recursively
- Strings → `document.createTextNode()`
- `null` / `undefined` → skipped (safe for conditional rendering)
- Everything else → `.toString()` wrapped in a text node

## `gui-layout` Slots

`gui-layout` provides a responsive grid layout with these slots:

| Slot | Purpose |
|---|---|
| `header` | Top bar (breadcrumbs, actions) |
| `navigation-header` | Above the nav tree |
| `navigation` | Navigation panel (typically `<gui-nav />`) |
| `navigation-footer` | Below the nav tree |
| `main-header` | Above main content |
| `main` | Primary content area |
| `main-footer` | Below main content |
| `aside` | Right sidebar |
| `footer` | Bottom bar |

The layout is responsive: below the breakpoint (default 920px), navigation and aside collapse into drawers with a hamburger menu. Set `breakpoint` property to customize.

### `nav.json`

Place in `public/nav.json`. `<gui-nav />` auto-discovers it by walking up from the current pathname.

```json
[
  { "label": "Dashboard", "route": "dashboard/" },
  {
    "label": "Reports",
    "route": "reports/",
    "link": true,
    "children": [
      { "label": "Monthly", "route": "monthly/" },
      { "label": "Annual", "route": "annual/" }
    ]
  }
]
```

- `link: true` makes a collapsible parent also clickable as a link.
- `expanded: true` opens the group by default.
- Child `route` values are relative to the parent `route`.

## Creating Custom Components

### Pattern 1: Plain Function (No Component)

For simple, non-reusable UI fragments, just use a function:

```tsx
function userCard(name: string, role: string): HTMLElement {
  return (
    <gui-card>
      <div slot="header">{name}</div>
      <p>{role}</p>
    </gui-card>
  ) as HTMLElement;
}

document.body.appendChild(userCard('Alice', 'Admin'));
```

### Pattern 2: GuiElement (Shadow DOM Component)

For reusable components with encapsulated styles:

```tsx
import { GuiElement, css, registerCustomElement } from '@greycat/web';
import style from './my-widget.css?inline';

export class MyWidget extends GuiElement {
  static override styles = [css(style)];

  constructor() {
    super();
    this.shadowRoot.appendChild(
      <div className="root">
        <slot />
      </div>,
    );
  }

  connectedCallback(): void {
    // Element is now in the DOM
    const signal = this.abortSignal(); // aborts on disconnect
    window.addEventListener('resize', this._onResize, { signal });
  }

  private _onResize = () => { /* ... */ };
}

registerCustomElement('my-widget', MyWidget);
```

Key features of `GuiElement`:
- Auto-creates Shadow DOM with adopted stylesheets
- `static styles` array is merged with the base component styles
- `addDisposable(fn)` — registers cleanup called on disconnect
- `abortSignal()` — returns an `AbortSignal` tied to element lifecycle
- `dispose()` is called automatically on `disconnectedCallback`

### Pattern 3: GuiValueElement (Reactive Value Component)

For components that render based on a data value:

```tsx
import { GuiValueElement, css, registerCustomElement } from '@greycat/web';
import style from './my-display.css?inline';

export class MyDisplay extends GuiValueElement<string[]> {
  static override styles = [css(style)];

  private _value: string[] = [];

  get value(): string[] { return this._value; }
  set value(v: string[]) {
    this._value = v;
    this._internalUpdate(); // queues batched update via queueMicrotask
  }

  override update(): void {
    this.shadowRoot.replaceChildren(
      <ul>
        {this._value.map((item) => <li>{item}</li>)}
      </ul>,
    );
  }
}

registerCustomElement('my-display', MyDisplay);
```

Key features of `GuiValueElement<T>`:
- Abstract `value` property triggers batched updates
- Multiple rapid `value` sets are coalesced into a single `update()` call
- `updateComplete` promise resolves after the next update completes
- `connectedCallback` auto-triggers the first update

### Registration Boilerplate

Every custom element **must** declare itself in both `HTMLElementTagNameMap` and `GreyCat.JSX.IntrinsicElements`.

`GreyCat.Element<T, EventMap>` accepts a second generic for **typed custom events**. This is how JSX knows which `on*` handlers are valid and what their `detail` type is:

```tsx
// Define a custom event map for your component
interface MyWidgetEventMap {
  'my-widget-select': CustomEvent<{ index: number; item: string }>;
  'my-widget-close': CustomEvent<void>;
}

declare global {
  // Merge into global HTMLElementEventMap so addEventListener is typed everywhere
  interface HTMLElementEventMap extends MyWidgetEventMap {}

  interface HTMLElementTagNameMap {
    'my-widget': MyWidget;
  }

  namespace GreyCat {
    namespace JSX {
      interface IntrinsicElements {
        // The second generic types the on* handlers in JSX
        'my-widget': GreyCat.Element<MyWidget, MyWidgetEventMap>;
      }
    }
  }
}
```

This enables fully typed JSX — both properties and events:

```tsx
<my-widget
  value={data}
  onmy-widget-select={(ev) => {
    // ev.detail is typed as { index: number; item: string }
    console.log(ev.detail.index);
  }}
/>
```

Without the event map, only standard `HTMLElementEventMap` events (`onclick`, `oninput`, etc.) would be typed.

## Events

### Custom Events

All `gui-*` events bubble and are composed (cross Shadow DOM boundaries):

| Event | When | Detail |
|---|---|---|
| `gui-input` | Real-time value change (as user types) | Current value |
| `gui-change` | Value committed (blur, Enter key) | Final value |
| `gui-submit` | Form submission | Submitted value |
| `gui-click` | Element clicked | Click data |
| `gui-dblclick` | Double-click | Click data |
| `gui-update` | Value updated programmatically | Updated value |

### JSX Event Handlers

The runtime strips the `on` prefix and calls `addEventListener`. This works for **any** event name, including hyphenated ones:

```tsx
// Standard DOM event
<button onclick={(ev) => console.log('clicked', ev)}>Click</button>

// gui-* event
<gui-table ongui-change={(ev) => console.log(ev.detail)} />

// Shoelace event
<sl-select onsl-change={(ev) => handleChange(ev)} />

// Table-specific events
<gui-table ongui-table-dblclick={(ev) => {
  const { rowIdx, colIdx } = ev.detail;
}} />
```

### Imperative Event Listeners

```ts
const table = document.createElement('gui-table');
table.addEventListener('gui-change', (ev) => {
  console.log('New value:', ev.detail);
});
```


Themes are controlled via `data-theme` attribute and `sl-theme-*` class on body:

```ts
import { toggleTheme, currentTheme } from '@greycat/web';
```

## Gotchas and Pitfalls

1. **No re-rendering.** Elements are created once (whether via JSX or `createElement`). To update, hold a reference and mutate:
   ```ts
   const table = document.createElement('gui-table');
   // later:
   table.value = newData;
   ```

2. **`gui-*` components require `gc.sdk.init()`** to complete first. They are registered inside `init()`. Using them before init creates unknown/unupgraded elements.

3. **Fragment children disappear.** A `DocumentFragment` empties when appended to the DOM. Don't store a fragment expecting to reuse its children.

4. **Always import `@greycat/web/greycat.css`** (or `greycat-full.css`). Without it, `gui-*` and `sl-*` components will render without proper styling. Never write aggressive CSS resets that conflict with these styles.

5. **Type casting JSX.** When using JSX, the return type is `Node`. Cast to the specific type for typed access:
   ```tsx
   const chart = (<gui-chart config={cfg} />) as GuiChart;
   chart.compute();
   chart.update();
   ```
   With `document.createElement()`, the return type is already correct thanks to `HTMLElementTagNameMap`.

6. **`project.d.ts` is auto-generated.** Never edit it. It provides TypeScript types for the `gc.project.*` namespace (your GreyCat backend functions). Generated by `greycat codegen ts`.

7. **CSS in Shadow DOM.** Components using `GuiElement` have Shadow DOM. External CSS won't penetrate. Use `static styles = [css(inlineStyle)]` with `?inline` CSS imports for component-scoped styles.

8. **MPA navigation is full page loads.** There is no client-side router. Each `.html` file under `app/` is a separate entry point. Use `<a href>` or `window.location` for navigation.

9. **Check GreyCat stdlib via LSP before workarounds.** Functions like `uniformf`, `normalf`, etc. may already exist in GreyCat's standard library. Always verify available functions before implementing custom solutions.
